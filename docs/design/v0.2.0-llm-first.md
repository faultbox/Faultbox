# Epic: v0.2.0 — LLM-First Fault Injection

## Vision

Make Faultbox the default fault injection tool for LLM agents writing
production microservices. An agent should be able to: write code → write
a spec → run tests → parse failures → fix code — in a fully automated loop
with no human intervention.

## Principles

1. **Structured over human-readable** — every output must be machine-parseable
2. **Zero-config where possible** — generate specs from existing infrastructure
3. **Containerized by default** — agents run in CI, not on bare metal
4. **Native tool integration** — MCP/function-calling, not CLI string parsing
5. **Actionable feedback** — don't just report failures, suggest fixes

## Steps (ordered: easy+important → hard+less important)

### Step 1: Structured JSON Test Output
**Priority:** P0 — blocks everything else
**Effort:** Small (2-3 days)

Add `--format json` flag to `faultbox test` that outputs structured results
to stdout (human output moves to stderr).

```json
{
  "version": "0.2.0",
  "file": "faultbox.star",
  "seed": 42,
  "summary": {"passed": 3, "failed": 1, "total": 4},
  "tests": [
    {
      "name": "test_write_failure",
      "status": "fail",
      "duration_ms": 420,
      "seed": 42,
      "error": {
        "message": "assert_eq failed: 200 != 503",
        "expected": 503,
        "actual": 200,
        "assertion": "assert_eq",
        "line": 12
      },
      "faults": [
        {
          "service": "db",
          "syscall": "write",
          "action": "deny",
          "errno": "EIO",
          "hits": 3,
          "label": "disk failure"
        }
      ],
      "trace": [
        {
          "seq": 1,
          "service": "db",
          "syscall": "write",
          "decision": "deny(EIO)",
          "path": "/tmp/inventory.wal",
          "timestamp_us": 1234567
        }
      ],
      "syscall_summary": {
        "db": {"total": 45, "faulted": 3, "breakdown": {"write": 20, "read": 15, "connect": 10}},
        "api": {"total": 30, "faulted": 0, "breakdown": {"write": 10, "connect": 5, "read": 15}}
      }
    }
  ]
}
```

**Why first:** Every subsequent step (MCP, diagnostics, CI integration) depends
on structured output. Without it, agents parse regex on console text.

**Acceptance criteria:**
- `faultbox test spec.star --format json` writes JSON to stdout
- Human output (logs, progress) goes to stderr
- All existing fields (trace, faults, syscall summary) included
- Error includes line number and assertion type
- `--format json` + `--runs N` outputs all runs

### Step 2: Docker Image
**Priority:** P0 — required for CI/agent use
**Effort:** Small (1-2 days)

Publish `ghcr.io/faultbox/faultbox:latest` and `ghcr.io/faultbox/faultbox:0.2.0`.

```dockerfile
FROM ubuntu:24.04
COPY faultbox faultbox-shim /usr/local/bin/
ENTRYPOINT ["faultbox"]
```

Build in release workflow alongside the tar.gz artifacts.

**Usage by agents:**
```bash
docker run --privileged ghcr.io/faultbox/faultbox test /workspace/faultbox.star
```

**Why privileged:** seccomp-notify requires `CAP_SYS_ADMIN` or `--privileged`.
Document this clearly — it's a hard requirement, not a convenience.

**Acceptance criteria:**
- Multi-arch image (linux/amd64 + linux/arm64)
- Published on every release tag
- `docker run ghcr.io/faultbox/faultbox --version` works
- Document in README and site
- GitHub Actions example in docs

### Step 3: `faultbox init --from-compose`
**Priority:** P1 — major DX for LLM agents
**Effort:** Medium (3-5 days)

Generate a starter `.star` spec from an existing `docker-compose.yml`:

```bash
faultbox init --from-compose docker-compose.yml > faultbox.star
```

**What it does:**
1. Parse compose services, ports, depends_on, healthchecks
2. Detect protocols from port conventions (5432→postgres, 6379→redis, etc.)
3. Generate service declarations with correct wiring
4. Generate a `test_happy_path()` skeleton per service
5. Generate `scenario()` registrations

**Example output:**
```python
# Generated from docker-compose.yml
api = service("api",
    image="myapp-api:latest",
    interface("http", "http", 8080),
    env={"DATABASE_URL": "postgres://db:5432/mydb"},
    depends_on=[db],
    healthcheck=http("localhost:8080/health"),
)

db = service("db",
    image="postgres:16",
    interface("pg", "postgres", 5432),
    healthcheck=tcp("localhost:5432"),
)

def test_happy_path():
    resp = api.http.get(path="/health")
    assert_eq(resp.status, 200)

scenario(test_happy_path)
```

**Why this order:** An LLM agent writing a service already has a compose file.
Generating the spec from it means the agent writes zero Faultbox boilerplate —
it goes straight to writing fault scenarios.

**Acceptance criteria:**
- Parses compose v3 format
- Maps well-known ports to protocols
- Generates healthchecks from compose healthcheck or port conventions
- Handles depends_on ordering
- `--format json` outputs spec as structured data (for LLM consumption)

### Step 4: MCP Server
**Priority:** P1 — native LLM tool integration
**Effort:** Medium (3-5 days)

`faultbox mcp` starts an MCP (Model Context Protocol) server that exposes
Faultbox as a tool for Claude, GPT, and other LLM agents.

**Tools exposed:**

| Tool | Description |
|------|-------------|
| `run_test` | Run a test file, return structured results |
| `run_single_test` | Run a specific test by name |
| `list_tests` | Discover tests in a .star file |
| `generate_faults` | Run failure generator on a spec |
| `get_trace` | Get detailed trace for a test run |
| `init_spec` | Generate spec from compose or binary |

**Example interaction:**
```
Agent → run_test(file="faultbox.star")
Faultbox → {tests: [{name: "test_write_failure", status: "fail", error: ...}]}
Agent → [reads error, fixes code, reruns]
```

**Transport:** stdio (standard for MCP — agent spawns `faultbox mcp` as subprocess).

**Implementation:** Reuse the JSON output from Step 1 internally. The MCP
server is a thin wrapper that maps tool calls to CLI commands.

**Acceptance criteria:**
- `faultbox mcp` starts MCP server on stdio
- All 6 tools implemented
- Works with Claude Code, Claude Desktop, Cursor
- Tool descriptions include parameter schemas
- Error responses are structured (not stack traces)

### Step 5: Structured Diagnostic Feedback
**Priority:** P2 — improves agent fix accuracy
**Effort:** Medium (3-5 days)

Add a `diagnostics` field to JSON output that gives actionable hints when
tests fail. This helps LLM agents decide *what to fix* without understanding
seccomp internals.

```json
{
  "diagnostics": [
    {
      "level": "error",
      "code": "FAULT_FIRED_BUT_SUCCESS",
      "message": "write fault fired 3 times on 'db' but API returned 200",
      "suggestion": "Service 'api' may not be checking write errors from 'db'. Look for missing error handling in the write/persist path.",
      "related_faults": ["write=deny(EIO) on db"],
      "related_trace": [1, 2, 3]
    },
    {
      "level": "warning",
      "code": "FAULT_INSTALLED_NOT_FIRED",
      "message": "fsync fault on 'db' was installed but never fired",
      "suggestion": "Service 'db' may use fdatasync instead of fsync, or may not sync at all. Check with --debug to see actual syscalls.",
      "related_faults": ["fsync=deny(EIO) on db"]
    }
  ]
}
```

**Diagnostic rules:**

| Code | Condition | Suggestion |
|------|-----------|------------|
| `FAULT_FIRED_BUT_SUCCESS` | Fault hit >0 times, test passed unexpectedly | Missing error handling |
| `FAULT_NOT_FIRED` | Fault installed, 0 hits | Wrong syscall variant or path |
| `ASSERTION_WRONG_VALUE` | assert_eq mismatch | Show expected vs actual with context |
| `SERVICE_CRASHED` | Exit code != 0 during test | Unhandled error caused panic/crash |
| `TIMEOUT_DURING_FAULT` | Test timed out while fault active | Possible deadlock or infinite retry |
| `MULTIPLE_FAULTS_INTERACTION` | >1 fault active, unexpected result | Cascading failure — simplify test |

**Why this order:** Diagnostics are most valuable when consumed by an LLM
through structured JSON (Step 1) or MCP (Step 4). Building it earlier would
mean formatting it for human text, then reformatting later.

**Acceptance criteria:**
- `diagnostics` array in JSON output for every test
- At least the 6 rules above implemented
- Suggestions reference service names and syscalls specifically
- Works in both `--format json` and MCP modes

### Step 6: GitHub Actions Integration
**Priority:** P2 — CI/CD for agent workflows
**Effort:** Small (2-3 days)

Publish `faultbox/test-action` GitHub Action:

```yaml
- uses: faultbox/test-action@v1
  with:
    spec: faultbox.star
    format: json
```

**What it does:**
1. Pulls the Docker image (Step 2)
2. Runs `faultbox test` with `--privileged`
3. Posts results as PR comment (pass/fail summary)
4. Uploads JSON trace as artifact
5. Fails the workflow if tests fail

**Why:** LLM agents (Devin, Claude Code, Cursor) create PRs. The action runs
Faultbox on every PR, giving the agent feedback without manual intervention.

**Acceptance criteria:**
- Action published on GitHub Marketplace
- Works with `--privileged` (document runner requirements)
- PR comment with pass/fail summary
- JSON artifact uploaded
- Example workflow in docs

### Step 7: `faultbox init --from-binary` (Auto-Discovery)
**Priority:** P3 — nice to have
**Effort:** Large (5-7 days)

Run a binary, observe its syscalls, and generate a spec from observed behavior:

```bash
faultbox init --from-binary ./my-service --port 8080
# Starts the service, sends probe requests, records syscall patterns
# Generates spec with observed interfaces, paths, and syscall profile
```

**What it does:**
1. Launch binary with trace-all seccomp filter
2. Send HTTP probes to discovered ports
3. Record syscall patterns (which files it writes, which sockets it opens)
4. Generate spec with realistic fault targets

**Why last:** Requires significant inference logic (guessing protocols from
syscall patterns). `--from-compose` covers the common case. This is the
"magic" version for when there's no compose file.

**Acceptance criteria:**
- Discovers HTTP endpoints via port probing
- Identifies file write patterns (WAL, data files, logs)
- Generates meaningful fault targets
- Works with Go, Java, Python, Rust, Node binaries
